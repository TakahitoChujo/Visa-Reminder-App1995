## 🎯 移行の概要

### 移行内容

| 項目 | 旧形式（v1） | 新形式（v2） |
|-----|-----------|------------|
| **暗号化方式** | XOR + SHA256 | AES-256-GCM |
| **セキュリティレベル** | 🔴 低（Known-plaintext攻撃に脆弱） | 🟢 高（NIST準拠） |
| **改ざん検出** | ❌ なし | ✅ あり（Authentication Tag） |
| **データ形式** | `"iv:ciphertext"`<br>（2フィールド） | `"v2:iv:ciphertext"`<br>（3フィールド） |
| **IVサイズ** | 16バイト（32文字Hex） | 12バイト（24文字Hex） |

### 移行対象データ

1. **在留カードのメモ**
   - テーブル: `residence_cards`
   - カラム: `memo`

2. **チェックリスト項目のメモ**
   - テーブル: `checklist_items`
   - カラム: `memo`

### 影響範囲

- **既存ユーザー**: 自動的にv1 → v2へ移行（透過的）
- **新規ユーザー**: 最初からv2形式で保存
- **データ損失リスク**: なし（既存データは保持）

## 🔄 移行戦略

### 採用する戦略: 段階的移行（Lazy Migration）

**方針**:
- データの**読み取り時**に自動的にv1 → v2へ変換
- 一括移行は行わない（アプリ起動時の遅延なし）
- ユーザーに移行を意識させない（透過的）

**メリット**:
- ✅ アプリ起動が遅くならない
- ✅ エラーハンドリングが容易
- ✅ ユーザー体験への影響が最小
- ✅ 移行失敗時もデータが失われない

**デメリット**:
- ⚠️ 移行完了までの期間が不明（ユーザーの利用頻度に依存）
- ⚠️ v1とv2が混在する期間がある

### 却下した戦略

#### 1. 一括移行（アプリ起動時）

**却下理由**:
- ❌ アプリ起動が遅くなる（大量データの場合）
- ❌ 移行失敗時の影響が大きい
- ❌ ユーザー体験が悪化

#### 2. バックグラウンド移行

**却下理由**:
- ❌ 実装が複雑
- ❌ バッテリー消費が増加
- ❌ React Nativeではバックグラウンド処理が制限される

## 🔀 移行フロー

### 自動移行のフロー図

```
ユーザーがデータを読み取る
  ↓
EncryptionService.decrypt() 呼び出し
  ↓
detectEncryptionVersion() でバージョン判定
  ↓
┌─────────────────┐
│ v2形式？        │
└─────────────────┘
  │ YES     │ NO
  ↓         ↓
  │    ┌─────────────────┐
  │    │ v1形式？        │
  │    └─────────────────┘
  │      │ YES     │ NO
  │      ↓         ↓
  │   decryptV1()  throw Error
  │      ↓
  │   平文を取得
  │      ↓
  │   encryptV2() で再暗号化
  │      ↓
  │   Repository層でDB更新
  │      ↓
  └──────┴─────────→ 平文を返却
            ↓
        移行完了
```

### コード例: Repository層

```typescript
/**
 * ResidenceCardRepository.getById()
 */
async getById(id: string): Promise<ResidenceCardDecrypted | null> {
  // 1. DBからデータ取得
  const card = await this.db.getResidenceCard(id);
  if (!card || !card.memo) return card as ResidenceCardDecrypted;

  // 2. バージョン判定
  const version = encryptionService.detectEncryptionVersion(card.memo);

  // 3. 復号化
  const decryptedMemo = await encryptionService.decrypt(card.memo);

  // 4. v1の場合、v2に移行
  if (version === 'v1') {
    const newEncrypted = await encryptionService.encrypt(decryptedMemo);
    await this.db.updateResidenceCard(id, { memo: newEncrypted });
    console.log(`[Migration] Card ${id}: v1 → v2 ✓`);
  }

  // 5. 復号化済みデータを返却
  return { ...card, memo: decryptedMemo };
}
```

### 移行の進捗モニタリング

```typescript
/**
 * 移行進捗を確認するヘルパー関数（開発/デバッグ用）
 */
async getMigrationProgress(): Promise<{
  totalRecords: number;
  v1Records: number;
  v2Records: number;
  progress: number; // 0-100%
}> {
  const cards = await this.db.getAllResidenceCards();
  let v1Count = 0;
  let v2Count = 0;

  for (const card of cards) {
    if (!card.memo) continue;

    const version = encryptionService.detectEncryptionVersion(card.memo);
    if (version === 'v1') v1Count++;
    if (version === 'v2') v2Count++;
  }

  const total = v1Count + v2Count;
  const progress = total > 0 ? Math.round((v2Count / total) * 100) : 100;

  return {
    totalRecords: total,
    v1Records: v1Count,
    v2Records: v2Count,
    progress,
  };
}
```

## 🔙 ロールバック手順

### ケース1: 移行中にバグが発見された場合

**状況**: AES-256-GCM実装にバグがあり、復号化に失敗する

**対応手順**:

1. **アプリのロールバック**
   ```bash
   # 旧バージョンのアプリを再デプロイ
   git checkout v1.0.0  # 移行前のバージョン
   npm run build
   npm run deploy
   ```

2. **データの確認**
   - v1データは保持されているため、旧バージョンで読み取り可能
   - v2に移行済みのデータは読み取れない（新規暗号化データのみ）

3. **ユーザーへの通知**
   ```
   件名: アプリアップデートのお知らせ

   平素よりご利用いただき、ありがとうございます。
   本日リリースしたバージョン1.1.0に不具合が発見されたため、
   一時的に旧バージョン1.0.0に戻させていただきました。

   ご迷惑をおかけし、申し訳ございません。
   ```

### ケース2: 緊急時の完全ロールバック

**状況**: 致命的な問題が発生し、すべてのデータをv1に戻す必要がある

**対応手順**:

1. **データベースバックアップから復元**
   ```sql
   -- バックアップから復元（移行前の状態）
   RESTORE DATABASE residence_reminder
   FROM BACKUP 'backup_2026-02-16.sql';
   ```

2. **暗号化サービスをv1に固定**
   ```typescript
   // 緊急フラグ（Remote Configで制御）
   const FORCE_V1_ENCRYPTION = true;

   public async encrypt(plaintext: string): Promise<string> {
     if (FORCE_V1_ENCRYPTION) {
       return this.encryptV1(plaintext); // XOR暗号化
     }
     return this.encryptV2(plaintext); // AES-256-GCM
   }
   ```

3. **ユーザーへの謝罪と補償**
   - アプリ内通知
   - メール通知（該当ユーザーのみ）
   - 必要に応じてプレミアム機能の無償提供

### ロールバックのリスク評価

| シナリオ | データ損失リスク | 影響範囲 | 復旧時間 |
|---------|---------------|---------|---------|
| **軽微なバグ** | なし | 一部機能のみ | 1-2時間 |
| **復号化失敗** | なし（v1データ保持） | 移行済みデータのみ | 4-6時間 |
| **完全ロールバック** | 移行後のデータ | 全ユーザー | 12-24時間 |

## ❓ FAQ

### Q1. 移行はいつ完了しますか？

**A**: ユーザーがデータを読み取ったタイミングで移行されるため、完了時期は不定です。
ただし、アプリを頻繁に使用するユーザーは1週間以内に移行が完了します。

**確認方法**:
```typescript
const progress = await this.getMigrationProgress();
console.log(`移行進捗: ${progress.progress}%`);
```

### Q2. 移行中にアプリがクラッシュしたらどうなりますか？

**A**: データは失われません。

**理由**:
- v1データは削除されない（移行成功後のみDB更新）
- トランザクション処理により、中途半端な状態にならない

**復旧**:
- アプリを再起動すれば、次回読み取り時に再度移行が試行される

### Q3. Web版のセキュリティは大丈夫ですか？

**A**: ネイティブアプリより安全性は低いですが、改善されています。

**改善内容**:
- ❌ 旧: localStorage（平文）
- ✅ 新: IndexedDB（XSSリスクは残る）

**推奨事項**:
- 機密性の高い情報はメモに記載しない
- Web版使用時はブラウザ拡張機能に注意

### Q4. 移行失敗時の通知はありますか？

**A**: エラーログに記録されますが、ユーザーには通知されません。

**理由**:
- 移行失敗してもv1形式で読み取り可能
- 次回アクセス時に再試行される

**モニタリング**:
```typescript
// エラーログを定期的に確認
console.error('[Migration Error]', error);
```

### Q5. キーをなくしたらどうなりますか？

**A**: データは復号化できなくなります。

**対策**:
- SecureStore（iOS/Android）は端末固有のキーチェーンに保存
- アプリ削除時のみキーが失われる
- バックアップ・復元機能は提供しない（セキュリティ優先）

**ユーザーへの案内**:
```
アプリを削除すると、暗号化キーが失われ、
メモを復元できなくなります。

重要な情報は他の方法でバックアップしてください。
```

### Q6. 暗号化キーはローテーションしますか？

**A**: 現時点では実装していません（将来実装予定）。

**将来の計画**:
- 設定画面に「暗号化キーを再生成」ボタンを追加
- すべてのデータを新キーで再暗号化
- セキュリティインシデント発生時に実行

### Q7. パフォーマンスへの影響は？

**A**: 無視できるレベルです。

**ベンチマーク**:
- 暗号化: 約5ms（100文字）
- 復号化: 約5ms（100文字）
- 移行処理: 約10ms（1件あたり）

**大量データの場合**:
- 100件の移行: 約1秒
- ただし、段階的移行のため一度に100件処理されることはない

### Q8. 既存データのバックアップは必要ですか？

**A**: 推奨しますが、必須ではありません。

**理由**:
- 移行失敗時も既存データは保持される
- ロールバックが容易

**バックアップ方法**:
```bash
# SQLiteデータベースのバックアップ
adb pull /data/data/com.yourapp/databases/residence_reminder.db backup.db
```

### Q9. テストはどの程度必要ですか？

**A**: 以下のテストを実施してください。

**必須テスト**:
- ✅ 新規データの暗号化（v2形式）
- ✅ v1データの復号化
- ✅ v1 → v2 自動移行
- ✅ Unicode文字の処理（日本語・絵文字）
- ✅ 改ざん検出

**推奨テスト**:
- ✅ 大量データのパフォーマンス
- ✅ Web版のIndexedDB動作
- ✅ エラーハンドリング
