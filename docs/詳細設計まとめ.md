## 1. 詳細設計の要約

### 1.1 共通データ契約

- **役割**: 資格タイプ（code）、リマインダーキー、チェックリスト状態・項目（item_id）のコードとラベルを単一の情報源として定義。
- **成果物**: [共通データ契約.md](共通データ契約.md)
- **内容**: work / spouse / permanent_prep / other の4資格タイプ、4種類のリマインダーキー、not_started / done の状態、資格タイプ別チェックリスト項目（work_01〜, spouse_01〜 等）。多言語キー方針・日付形式・保持しないデータ（機微情報）を明記。

### 1.2 DB設計

- **役割**: クライアント側のローカル保存と、サーバー側（将来のアカウント連携・同期）のテーブル設計。
- **成果物**: [DB設計書.md](DB設計書.md)
- **内容**:
  - **クライアント**: residence（1件）、reminder_settings（4キーの boolean）、checklist_progress（item_id + status + memo）。マスタは共通契約を静的データとして保持。
  - **サーバー**: users, residence, reminder_settings, checklist_progress, device_tokens。個人を特定するカラムは持たない。ER 図とカラム定義を記載。

### 1.3 API設計

- **役割**: アカウント連携時・将来の同期用 REST API。MVP はオフライン優先のため、クライアントはローカルを主とし、オンライン時に同期する想定。
- **成果物**: [API設計書.md](API設計書.md)
- **内容**: GET/PUT /me/residence, GET/PUT /me/reminder-settings, GET/PUT /me/checklist-progress（および PATCH で項目別更新）、GET /master/qualification-types, GET /master/checklist-templates/:qualificationTypeId, POST /me/device-tokens。リクエスト／レスポンスの JSON 例、エラー・HTTP ステータス方針を記載。

### 1.4 UI・デザイン設計

- **役割**: 5画面の構成・データ対応、コンポーネント、デザイン仕様。
- **成果物**: [UI設計書.md](UI設計書.md)、[デザイン仕様.md](デザイン仕様.md)
- **内容**:
  - **UI設計書**: ホーム／登録・編集／リマインダー設定／チェックリスト／設定の画面別構成、表示要素と API/ローカル DB の対応、画面遷移図、コンポーネント一覧、オフライン・同期の注記。
  - **デザイン仕様**: トーン、カラー、タイポグラフィ、スペーシング、コンポーネントの見た目、アクセシビリティ、アイコン方針。

## 2. 整合性確認結果

### 2.1 資格タイプ・リマインダーキー・チェック項目ID の一致

| 項目 | 共通契約 | DB設計 | API設計 | UI設計 |
|------|----------|--------|---------|--------|
| 資格タイプ（code） | work, spouse, permanent_prep, other | qualification_type で同一コード | qualification_type で同一 | 共通契約のラベル/i18n_key で表示 |
| リマインダーキー | 4_months_before 等4種 | カラム名として同一 | キー名として同一 | 共通契約のラベルで表示 |
| チェックリスト状態 | not_started, done | status で同一 | status で同一 | チェック ON/OFF で対応 |
| チェック項目（item_id） | work_01, spouse_01 等 | item_id で同一 | item_id で同一 | テンプレートの item_id で紐づけ |

**結果**: 共通契約・DB・API・UI の4領域でコード・ID・キーが一致している。多言語キーも共通契約で定義され、UI で参照する方針で揃っている。

### 2.2 画面で必要なデータが API/DB で賄えるか

| 画面 | 必要データ | 提供元 |
|------|------------|--------|
| ホーム | 有効期限、残り日数、資格タイプ名、次回リマインド日、申請可能時期の目安 | GET /me/residence, GET /me/reminder-settings。残り日数・次回リマインド日はクライアントで expires_on から算出。資格タイプ名はマスタまたは共通契約。 |
| 登録・編集 | 有効期限、資格タイプ一覧、申請可能時期の目安 | PUT/GET /me/residence。資格タイプは GET /master/qualification-types またはローカルマスタ。 |
| リマインダー設定 | 4種の ON/OFF | GET/PUT /me/reminder-settings またはローカル reminder_settings。 |
| チェックリスト | 項目一覧（ラベル）、各項目の状態・メモ | GET /master/checklist-templates/:qualificationTypeId, GET/PUT /me/checklist-progress またはローカル。 |
| 設定 | 通知、規約・ポリシー、お問い合わせ | 静的画面。通知設定は OS またはリマインダー設定へのリンク。 |

**結果**: 各画面の表示・入力に必要なデータは、API のレスンスおよびローカル DB（共通契約のマスタ含む）で全て賄える。

### 2.3 個人を特定するデータを保存していないか

- **共通契約**: 保持しないデータとして在留カード番号、氏名、生年月日、住所等を明記。
- **DB設計**: residence は qualification_type, expires_on, created_at, updated_at のみ。users は external_id のみで、メール・氏名は持たない設計。
- **API**: 在留資格・リマインダー・チェックリストのペイロードに機微情報は含めない。
- **UI**: 機微情報を入力・表示する項目は定義していない。

**結果**: 要件 6.2（個人を特定し得る機微情報を取得・保存しない）を満たす設計になっている。

## 3. 次のステップ：実装タスク分解の案

詳細設計を前提に、実装フェーズで進めやすいタスク案をまとめる。

### 3.1 共通・基盤

- プロジェクト初期化（クロスプラットフォーム or ネイティブの選定、リポジトリ、環境）
- 共通データ契約のコード化（資格タイプ・リマインダーキー・チェックリストテンプレートの定数または JSON の配置）
- 日付フォーマット・「残り日数」「次回リマインド日」計算のユーティリティ

### 3.2 データ層

- クライアント側: ローカルストアの選定（SQLite / Realm / AsyncStorage 等）とスキーマ実装（residence, reminder_settings, checklist_progress）
- マスタデータの組み込み（共通契約の内容をアプリ内リソースとして配置）
- （将来）サーバー側: DB マイグレーション、テーブル作成

### 3.3 API・同期（オプション）

- 認証プレースホルダの実装
- /me/residence, /me/reminder-settings, /me/checklist-progress のクライアント SDK または API クライアント
- マスタ取得 API の呼び出し（またはオフライン専用の場合はマスタをバンドルのみで対応）
- （将来）オフライン編集のキューとオンライン復帰時の同期

### 3.4 UI・画面

- デザイン仕様に基づくテーマ・コンポーネント（カラー、タイポ、カード、ボタン、トグル、チェックボックス等）
- ホーム: 未登録／登録済みの2状態、カード表示、CTA
- 登録・編集: 日付ピッカー、資格タイプ選択、申請可能時期の目安表示、保存
- リマインダー設定: 4トグル、通知許可への導線
- チェックリスト: テンプレートに基づく項目リスト、チェック・メモ、外部リンク
- 設定: 通知、利用規約・プライバシーポリシー、お問い合わせ（課金はラベルのみ可）

### 3.5 通知

- ローカル通知（プッシュ）のスケジュール: expires_on と reminder_settings に基づき、4ヶ月前・3ヶ月前・1ヶ月前・2週間前の日付に通知を登録
- 通知タップ時のディープリンク（該当画面へ遷移）
- （将来）サーバー側でのプッシュ送信と POST /me/device-tokens の利用

### 3.6 品質・リリース

- 単体テスト（日付計算、マスタの整合性、保存・取得）
- E2E または画面単位のテスト（登録〜チェックリスト表示までの流れ）
- 利用規約・プライバシーポリシーの文案と表示
- ストア申請用の説明文・スクリーンショット準備
